# AGENTS.md

작업을 시작하기 전에 요구사항이 모호하거나 확인이 필요한 사항이 있으면, 진행 전에 사용자에게 질문한다.

## 사용자 운영 원칙

### 1. 사용자 판단이 필요한 것은 먼저 묻는다

- 기본 원칙은 `판단이 필요한 선택은 사용자에게 먼저 묻기`다.
- 예외는 사용자가 `개입 없이 끝까지 진행하라`고 명시한 경우다.
- 이 예외 상황에서는 Codex가 사용자를 대신해 임시 판단을 내려 진행할 수 있다.

### 2. 사용자 대신 내린 판단은 임시 기록만 남긴다

- 사용자 대신 내린 판단 중 `사용자가 직접 판단했을 법한 것`만 [codex-log.md](./codex-log.md)에 적는다.
- `codex-log.md`는 작업 일지가 아니라 `미결정 판단 큐`다.
- 사용자가 검토를 끝낸 항목은 삭제한다.
- 자잘한 구현 디테일, 단순 버그 수정, 자연스러운 후속 조치는 기록하지 않는다.

### 3. 차트 없는 보고서는 불충분하다

- 실험 보고서는 Markdown으로 작성한다.
- 보고서에는 반드시 `실제 차트 이미지`를 포함한다.
- 텍스트 표만으로 끝내지 않는다.
- 가능하면 차트 종류를 다양하게 넣는다.
- 보고서는 고등학생도 읽을 수 있게 각 표와 차트의 의미를 설명한다.
- 프롬프트 원문, 비교 사례, 대표 샘플도 가능한 한 함께 넣는다.

### 4. 실험 결과는 사람이 비교할 수 있어야 한다

- 가능하면 데이터셋 대표 샘플 이미지를 보고서에 넣는다.
- 가능하면 baseline 실패 / optimized 성공 같은 비교 사례를 넣는다.
- 프롬프트별 결과 차이를 사람이 눈으로 검토할 수 있어야 한다.

### 5. 보고서는 `docs/` 아래에 번호를 붙여 관리한다

- 실험 보고서는 루트가 아니라 `docs/` 아래에 둔다.
- 보고서 파일명은 `001_...`, `002_...`처럼 순번을 붙여 누적한다.
- 차트 이미지와 샘플 이미지는 보고서 번호에 맞는 `docs/assets/<번호>/` 아래에 둔다.
- 보고서는 길기만 한 작업 로그가 아니라, 핵심 결과와 해석이 빠르게 보이도록 구성한다.
- 각 섹션은 `무엇을 보여주는지`, `왜 중요한지`, `어떻게 해석해야 하는지`를 분명히 적는다.

### 5-1. 최적화 보고서는 라운드별 전체 후보를 숨기지 않는다

- optimizer 실험 보고서에는 각 round별 `전체 프롬프트 후보`와 각 후보의 성능 지표를 반드시 넣는다.
- winner만 요약 표에 적고 끝내지 않는다.
- 각 round 표에는 최소한 `start prompt`, `생성된 모든 candidate`, `winner 표시`가 있어야 한다.
- winner는 굵게 표시하거나 색상 등으로 눈에 띄게 강조한다.
- 본문이 길어지면 마지막 부록에라도 round별 전체 후보 표와 프롬프트 원문을 넣는다.
- 보고서만 읽어도 `어떤 프롬프트들이 검토되었고 왜 winner가 됐는지` 추적 가능해야 한다.

### 6. 환경과 버전 관리는 `uv` 기준으로 유지한다

- Python 패키지와 가상환경 관리는 `uv`를 사용한다.
- 새 의존성, 실행 방식, 테스트 방식도 `uv` 기준으로 맞춘다.

### 7. 커밋은 의미 있는 단위로 남긴다

- 작업 중간중간 의미 있는 변경 단위로 커밋한다.
- 여러 개념을 한 커밋에 무리하게 섞지 않는다.
- 사용자의 수정이 포함돼야 하는 경우, 요청이 있었다면 함께 커밋한다.

### 8. 정확도에 영향 줄 수 있는 전처리는 검증한다

- 이미지 리사이즈, padding, chunking 같은 전처리는 정확도에 영향이 있을 수 있다.
- 이런 변경은 적용 후 실제 샘플이나 지표로 품질 문제가 없는지 확인한다.
- 특히 `dimension` 축소처럼 정보를 잃을 수 있는 변경은 검증 없이 밀어넣지 않는다.

### 9. 기존 패턴보다 프로젝트 맥락을 우선한다

- 영수증 OCR, 문서 OCR, line OCR처럼 데이터 성격이 다르면 같은 전처리나 같은 실험 설정을 그대로 가정하지 않는다.
- 데이터셋 성격이 바뀌면 속도, 품질, 평가 방식이 달라질 수 있음을 먼저 점검한다.

### 10. 다중 Codex CLI 작업은 `worktree + branch` 분리를 기본으로 한다

- 여러 Codex CLI가 동시에 작업할 때는 가능하면 각자 별도 `git worktree`와 별도 브랜치에서 작업한다.
- `git worktree`는 작업 디렉터리 충돌을 줄여주지만, merge 충돌까지 없애주지는 않는다고 가정한다.
- 따라서 Codex는 다른 작업자의 변경을 덮어쓰지 않도록 하고, 충돌은 최종 `merge` 또는 `rebase` 시점에 해결하는 것을 기본 원칙으로 삼는다.
- 산출물 경로 분리를 억지로 강제하기보다, 브랜치 목적을 분명히 하고 커밋 단위를 명확히 유지하는 쪽을 우선한다.
- 이미 다른 Codex가 손댄 파일과 직접 충돌할 가능성이 높으면, 먼저 현 상태를 읽고 충돌 범위를 파악한 뒤 작업한다.
