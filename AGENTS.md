# AGENTS.md

작업을 시작하기 전에 요구사항이 모호하거나 확인이 필요한 사항이 있으면, 진행 전에 사용자에게 질문한다.

## 사용자 판단과 협업

### 사용자 판단이 필요한 것은 먼저 묻는다

- 기본 원칙은 `판단이 필요한 선택은 사용자에게 먼저 묻기`다.
- 예외는 사용자가 `개입 없이 끝까지 진행하라`고 명시한 경우다.
- 이 예외 상황에서는 Codex가 사용자를 대신해 임시 판단을 내려 진행할 수 있다.

### 사용자 대신 내린 판단은 임시 기록만 남긴다

- 사용자 대신 내린 판단 중 `사용자가 직접 판단했을 법한 것`만 [codex-log.md](./codex-log.md)에 적는다.
- `codex-log.md`는 작업 일지가 아니라 `미결정 판단 큐`다.
- 사용자가 검토를 끝낸 항목은 삭제한다.
- 자잘한 구현 디테일, 단순 버그 수정, 자연스러운 후속 조치는 기록하지 않는다.

### 다중 Codex CLI 작업은 `worktree + branch` 분리를 기본으로 한다

- 여러 Codex CLI가 동시에 작업하게 되니, 각자 별도 `git worktree`와 별도 브랜치에서 작업한다.
- `git worktree`는 작업 디렉터리 충돌을 줄여주지만, merge 충돌까지 없애주지는 않는다고 가정한다.
- 따라서 Codex는 다른 작업자의 변경을 덮어쓰지 않도록 하고, 충돌은 최종 `merge` 또는 `rebase` 시점에 해결하는 것을 기본 원칙으로 삼는다.
- 산출물 경로 분리를 억지로 강제하기보다, 브랜치 목적을 분명히 하고 커밋 단위를 명확히 유지하는 쪽을 우선한다.
- 이미 다른 Codex가 손댄 파일과 직접 충돌할 가능성이 높으면, 먼저 현 상태를 읽고 충돌 범위를 파악한 뒤 작업한다.

## 실험과 보고서

### 차트 없는 보고서는 불충분하다

- 실험 보고서는 Markdown으로 작성한다.
- 보고서에는 반드시 `실제 차트 이미지`를 포함한다.
- 텍스트 표만으로 끝내지 않는다.
- 가능하면 차트 종류를 다양하게 넣는다.
- 보고서는 고등학생도 읽을 수 있게 각 표와 차트의 의미를 설명한다.
- 프롬프트 원문, 비교 사례, 대표 샘플도 가능한 한 함께 넣는다.

### 실험 결과는 사람이 비교할 수 있어야 한다

- 가능하면 데이터셋 대표 샘플 이미지를 보고서에 넣는다.
- 가능하면 baseline 실패 / optimized 성공 같은 비교 사례를 넣는다.
- 프롬프트별 결과 차이를 사람이 눈으로 검토할 수 있어야 한다.

### 보고서는 `docs/` 아래에 번호를 붙여 관리한다

- 실험 보고서는 루트가 아니라 `docs/` 아래에 둔다.
- 보고서 파일명은 `001_...`, `002_...`처럼 순번을 붙여 누적한다.
- 차트 이미지와 샘플 이미지는 보고서 번호에 맞는 `docs/assets/<번호>/` 아래에 둔다.
- 보고서는 길기만 한 작업 로그가 아니라, 핵심 결과와 해석이 빠르게 보이도록 구성한다.
- 각 섹션은 `무엇을 보여주는지`, `왜 중요한지`, `어떻게 해석해야 하는지`를 분명히 적는다.

### 최적화 보고서는 라운드별 전체 후보를 숨기지 않는다

- optimizer 실험 보고서에는 각 round별 `전체 프롬프트 후보`와 각 후보의 성능 지표를 반드시 넣는다.
- winner만 요약 표에 적고 끝내지 않는다.
- 각 round 표에는 최소한 `start prompt`, `생성된 모든 candidate`, `winner 표시`가 있어야 한다.
- winner는 굵게 표시하거나 색상 등으로 눈에 띄게 강조한다.
- 본문이 길어지면 마지막 부록에라도 round별 전체 후보 표와 프롬프트 원문을 넣는다.
- 보고서만 읽어도 `어떤 프롬프트들이 검토되었고 왜 winner가 됐는지` 추적 가능해야 한다.

## 구현과 품질

### 환경과 버전 관리는 `uv` 기준으로 유지한다

- Python 패키지와 가상환경 관리는 `uv`를 사용한다.
- 새 의존성, 실행 방식, 테스트 방식도 `uv` 기준으로 맞춘다.

### 커밋은 의미 있는 단위로 남긴다

- 작업 중간중간 의미 있는 변경 단위로 커밋한다.
- 여러 개념을 한 커밋에 무리하게 섞지 않는다.
- 사용자의 수정이 포함돼야 하는 경우, 요청이 있었다면 함께 커밋한다.
- 커밋 메시지는 `제목 1줄`로 끝내지 말고, 가능하면 `본문`도 함께 적는다.
- 이 프로젝트에서는 커밋 메시지의 `제목`과 `본문`을 모두 한글로 적는다.
- 제목은 변경의 핵심을 한 줄로 요약하고, 본문에는 `무엇을 바꿨는지`, `왜 바꿨는지`, `검증은 어떻게 했는지`를 짧게 남긴다.
- 문서, 테스트, 스크립트, 버그 수정처럼 성격이 다른 변경이 함께 들어가면, 본문에서 항목을 나눠 설명한다.

### 정확도에 영향 줄 수 있는 전처리는 검증한다

- 이미지 리사이즈, padding, chunking 같은 전처리는 정확도에 영향이 있을 수 있다.
- 이런 변경은 적용 후 실제 샘플이나 지표로 품질 문제가 없는지 확인한다.
- 특히 `dimension` 축소처럼 정보를 잃을 수 있는 변경은 검증 없이 밀어넣지 않는다.

### 기존 패턴보다 프로젝트 맥락을 우선한다

- 영수증 OCR, 문서 OCR, line OCR처럼 데이터 성격이 다르면 같은 전처리나 같은 실험 설정을 그대로 가정하지 않는다.
- 데이터셋 성격이 바뀌면 속도, 품질, 평가 방식이 달라질 수 있음을 먼저 점검한다.

## Codex 작업 방식

### Codex 작업 요청은 `목표 + 제약 + 검증 + 완료조건`으로 준다

- Codex에게 일을 시킬 때는 최소한 아래 4가지를 한 번에 준다.
- `목표`: 무엇을 바꾸고 싶은지 한 문장으로 적는다.
- `제약`: 바꾸면 안 되는 API, 성능 우선순위, 의존성 추가 금지 같은 제한을 적는다.
- `검증`: 작업 후 반드시 실행할 명령을 적는다. 이 프로젝트에서는 가능하면 `uv run ...` 형태로 준다.
- `완료조건`: 어떤 상태가 되면 끝난 것으로 볼지 적는다.
- 예시: `src/glm_ocr_prompt_optimization/metrics.py`의 CER 계산 버그를 고치고, 공개 함수 시그니처는 유지하고, \`uv run pytest -q tests/test_metrics.py\` 로 검증하고, 실패 케이스 회귀 테스트가 추가되면 완료.`

### Codex는 탐색, 수정, 검증 순서로 움직인다

- Codex는 먼저 관련 파일, 테스트, 실행 경로를 찾고 나서 수정한다.
- 여러 파일을 건드리는 작업이면 수정 전에 `어떤 파일을 왜 바꾸는지`, `어떤 검증을 돌릴지`를 먼저 짧게 공유한다.
- 수정 후에는 최소 1개의 검증 명령을 실제로 실행한다.
- 검증을 못 돌렸다면 이유를 분명히 적고, 사용자가 이어서 실행할 정확한 명령을 남긴다.

### 요청이 넓으면 단계로 쪼개서 시킨다

- 큰 작업을 한 번에 `다 바꿔라`로 주지 말고 `영향 범위 조사 -> 최소 수정 -> 보고서/후속 정리` 순서로 나눈다.
- optimizer, dataset, reporting을 동시에 건드리는 요청이면 우선순위를 정해준다.
- 예시: `먼저 heldout benchmark split 로직만 고치고, 보고서 생성 스크립트 수정은 그 다음 단계로 미뤄라.`

### 보고 형식도 미리 지정한다

- 사용자는 필요하면 Codex에게 최종 보고 형식을 같이 지시한다.
- 예시: `최종 답변에는 변경 파일, 실행한 검증 명령, 남은 리스크만 짧게 적어라.`
- 리뷰를 맡길 때는 `버그/리스크/테스트 누락 위주로 보고하고 요약은 마지막에 짧게`라고 명시한다.

## 프로젝트 표준 실행 규칙

### 검증은 빠른 체크와 전체 체크를 분리한다

- 빠른 체크가 필요할 때는 `./scripts/codex_preflight.sh`를 우선 사용한다.
- 전체 회귀 확인이 필요할 때는 `./scripts/codex_preflight.sh --full`을 사용한다.
- CLI 진입점까지 확인해야 하면 `./scripts/codex_preflight.sh --cli`를 사용한다.
- `pytest` marker는 `unit`, `integration` 두 가지로 유지한다.
- 빠른 수정이면 `unit`, CLI나 runner 흐름까지 건드렸으면 `integration`도 포함한다.
- 사용자는 Codex에게 `빠른 체크만`, `전체 체크까지`, `CLI 포함` 중 무엇을 원하는지 명시하면 좋다.

### 자주 쓰는 실험 흐름은 스크립트로 고정한다

- 과거 보고서 기준으로 반복된 흐름은 세 가지다.
- `기존 manifest 실험` -> `./scripts/run_manifest_experiment.sh`
- `HF dataset 준비 후 실험` -> `./scripts/run_hf_prepare_and_experiment.sh`
- `held-out benchmark 생성 후 실험` -> `./scripts/run_heldout_benchmark_experiment.sh`
- 새 보고서 스크립트를 만들 때도 가능하면 `runs/<run-name>/seed`, `optimize`, `validation` 구조를 그대로 사용한다.

### 자연어 별칭을 스크립트와 연결해 해석한다

- 새 Codex CLI는 아래 표현을 보면 대응 스크립트를 우선 떠올린다.
- `manifest run`, `기존 manifest로 돌려라`, `이미 있는 dev/val manifest로 실험해라`
  -> `./scripts/run_manifest_experiment.sh`
- `hf prepare + run`, `HF 데이터셋 받아서 split 후 실험해라`, `AbdullahRian 데이터셋으로 바로 실험해라`
  -> `./scripts/run_hf_prepare_and_experiment.sh`
- `held-out benchmark + run`, `manifest 여러 개 합쳐 held-out로 다시 돌려라`, `held-out split 다시 만들고 실험해라`
  -> `./scripts/run_heldout_benchmark_experiment.sh`
- `빠른 체크`, `quick check`, `unit만 확인`
  -> `./scripts/codex_preflight.sh`
- `CLI 포함 체크`, `cli check`, `integration까지`
  -> `./scripts/codex_preflight.sh --cli`
- `전체 체크`, `full check`, `전체 회귀`
  -> `./scripts/codex_preflight.sh --full`
- 사용자가 위 표현 중 하나를 쓰면, Codex는 가능하면 해당 스크립트를 먼저 제안하거나 바로 사용한다.
- 요구사항이 그 스크립트의 입력 형식과 맞지 않으면 왜 안 맞는지 먼저 짧게 설명한다.
- 사용자가 모호하게 지시했거나 실제 실행 전에 계획 확인이 필요하면 각 스크립트에 `--dry-run`을 먼저 붙여 실행 계획을 보여줄 수 있다.

### 모호한 실험 요청도 기본 규칙으로 해석한다

- 사용자가 단지 `이 데이터로 실험 돌려`, `이걸로 평가해`, `실험 한번 해봐`처럼 말해도 Codex는 아래 순서로 입력 형태를 보고 기본 스크립트를 고른다.
- `dev.jsonl`와 `val.jsonl` 경로가 이미 주어졌거나, 둘 다 저장소 안에 명확히 존재하면
  -> `./scripts/run_manifest_experiment.sh`
- Hugging Face dataset id가 주어졌거나, `AbdullahRian/...` 같은 dataset 이름만 던져졌으면
  -> `./scripts/run_hf_prepare_and_experiment.sh`
- manifest가 여러 개 주어졌고 `합쳐서`, `held-out`, `다시 split`, `benchmark 다시 만들기` 같은 뜻이 있으면
  -> `./scripts/run_heldout_benchmark_experiment.sh`
- 위 셋 중 어디에도 분명히 안 맞으면 Codex는 바로 실행하지 말고, `이미 dev/val manifest가 있는지`, `HF에서 받아야 하는지`, `held-out를 다시 만들지`만 짧게 확인한다.
- 사용자가 명시적으로 `개입 없이 끝까지 진행하라`고 했으면, Codex는 저장소 안의 가장 가까운 manifest와 기존 run 패턴을 보고 임시 판단해 진행할 수 있다.
- 입력 해석이 맞는지 불안하면 실제 실행보다 먼저 `--dry-run`으로 확인하는 편을 우선한다.

### 코드 수정 시 기본 검증도 자동 선택한다

- 사용자가 검증 명령을 따로 안 줘도, Codex는 수정한 파일을 기준으로 최소 검증을 자동 선택해 실행한다.
- 아래 파일만 바뀌면 기본값은 `./scripts/codex_preflight.sh`다.
- 대상: `src/glm_ocr_prompt_optimization/metrics.py`, `dataset.py`, `optimizer.py`, `models.py`, `prompts.py`, `config.py`
- 아래 파일 중 하나라도 바뀌면 기본값은 `./scripts/codex_preflight.sh --cli`다.
- 대상: `src/glm_ocr_prompt_optimization/cli.py`, `experiment.py`, `ocr_client.py`, `arize_logger.py`, `logger.py`
- `scripts/` 아래 실행 스크립트를 바꾸면 기본값은 `bash -n <script>`와 `./scripts/codex_preflight.sh --cli`다.
- `docs/`나 보고서 Markdown만 바뀌면 테스트는 필수가 아니지만, 관련 스크립트를 같이 바꿨으면 그 스크립트 문법 검사는 실행한다.
- 여러 계층을 동시에 바꿨거나 판단이 애매하면 더 약한 검증이 아니라 더 강한 검증을 고른다.
- 사용자가 `빠른 체크만`이라고 제한하지 않는 한, Codex는 최소 검증을 실제로 실행한 뒤 결과를 답변에 적는다.
