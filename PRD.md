# PRD: GLM-OCR 한국어 OCR 정확도 향상을 위한 Prompt Optimization

## 1. 문서 목적

이 프로젝트의 목적은 **GLM-OCR의 기본 프롬프트 `Text Recognition:`을 출발점으로 삼아, 한국어 중심 영수증 이미지에 대해 plain text OCR 정확도를 높이는 최적 프롬프트를 찾는 것**이다. 실험 대상은 JSON 추출이 아니라 **문자 인식 정확도 자체**이며, 출력 포맷도 GLM-OCR가 기본적으로 생성하는 plain text를 유지한다. GLM-OCR는 vLLM 배포를 공식 지원하며, 공식 예시 역시 chat/messages 형태로 이미지와 `"Text Recognition:"` 텍스트를 함께 넣는 방식을 사용한다. ([Hugging Face][1])

## 2. 문제 정의

현재 baseline인 `Text Recognition:`만 사용할 경우, 실제 사용상 중요한 실패 패턴이 세 가지다. 첫째, **중국어가 섞여 나오는 비한국어 출력**이다. 둘째, **의미 없는 반복 텍스트**가 길게 생성되는 경우다. 셋째, **애매한 오타**로 인해 사람이 보기엔 거의 맞지만 실제 업무상 오류가 되는 경우다. 따라서 이 프로젝트는 단순히 “문자가 얼마나 비슷한가”만 보는 OCR 실험이 아니라, **한국어 전사 안정성까지 포함한 OCR 실험**으로 설계해야 한다.

## 3. 성공 기준

주평가지표는 **CER(Character Error Rate)** 로 한다. 이유는 OCR 연구에서 가장 널리 쓰이는 표준 지표이고, 한국어 영수증 OCR 벤치마크 KORIE 역시 OCR 성능 비교에서 CER를 핵심 지표로 사용했기 때문이다. KORIE 논문에서는 PaddleOCR가 OCR baseline 중 가장 낮은 CER 15.84%를 기록했다고 보고한다. ([MDPI][2])

다만 이번 프로젝트의 실제 실패 패턴은 CER 하나로 충분히 포착되지 않는다. 예를 들어 긴 반복 생성이나 중국어 혼입은 CER에만 맡기면 과소평가될 수 있다. 그래서 성공 기준은 아래 두 층으로 둔다.

첫 번째는 **주지표**로서의 CER다.
두 번째는 **운영상 패널티 지표**다. 여기에는 비한국어 혼입률, 반복 생성률, 공백 출력률을 둔다.

최종 성공 조건은 다음처럼 둔다.

* baseline `Text Recognition:` 대비 **CER 10% 이상 상대 개선**
* 비한국어 혼입 사례 비율 **50% 이상 감소**
* 의미 없는 반복 출력 사례 비율 **50% 이상 감소**
* 위 세 조건 중 하나만이 아니라, **CER 개선과 안정성 개선이 함께 성립**해야 채택

이 기준은 “오타만 조금 줄인 프롬프트”보다 “실제로 덜 망가지는 프롬프트”를 선택하기 위한 장치다.

## 4. 대상 데이터셋

주 데이터셋은 **KORIE**로 한다. KORIE는 2026년에 공개된 한국어 영수증 벤치마크로, 748장의 한국 소매 영수증을 포함하고, OCR 평가를 위한 전사 라벨과 분할 정보를 함께 제공한다. 논문은 KORIE가 한국어 영수증의 열악한 열감지 인쇄 특성, 밴딩, 접힘, 저대비 등 현실적 노이즈를 반영한다고 설명한다. 이번 프로젝트의 목표가 “한국어 중심 OCR 정확도 향상”이므로, 영어 중심의 SROIE보다 KORIE가 1차 벤치마크로 더 적합하다. ([MDPI][2])

보조 검증용 데이터셋은 **SROIE를 선택하지 않는다.** 이유는 당신의 목표가 GLM-OCR의 일반 OCR 모델 우수성 증명이 아니라 **한국어 영수증에 대한 practical prompt optimization**이기 때문이다. 데이터셋을 늘리면 일반화 검증에는 좋지만, 현재는 예산과 시간보다 **프롬프트 탐색 효율**이 더 중요하다.

실험 데이터 분할은 다음처럼 둔다.

* 개발용 샘플셋: KORIE에서 60장
* 최종 검증셋: KORIE에서 100장
* 나머지는 이번 실험에서 사용하지 않음

이렇게 작게 가져가는 이유는 예산 제한과 optimizer 반복 비용 때문이다. Arize의 Prompt Learning 문서도 dataset quality와 iterative improvement를 강조하지, 대규모 데이터가 필수라고 말하지는 않는다. 초기 최적화는 작은 고품질 셋으로 시작하는 것이 맞다. ([Arize AI][3])

## 5. 모델 및 시스템 제약

OCR task model은 **GLM-OCR 로컬 서빙**으로 한다. 배포는 vLLM 공식 예시를 따른다. Hugging Face README는 `vllm serve zai-org/GLM-OCR --allowed-local-media-path / --port 8080` 예시를 제공하며, vLLM recipes 문서는 MTP speculative decoding도 지원한다고 안내한다. 다만 GTX 1660 Super 환경에서는 속도와 안정성을 우선해야 하므로, **이번 PRD에서는 speculative decoding 최적화는 제외**하고 기본 서빙으로 시작한다. ([Hugging Face][1])

optimizer 및 meta-prompting용 LLM은 **gpt-5-nano**로 한다. OpenAI 공식 가격 문서는 gpt-5-nano의 Batch API 기준 입력 $0.05/1M tokens, 출력 $0.40/1M tokens라고 안내한다. 따라서 소규모 샘플셋에 대해 프롬프트 생성·오류 요약·후보 리라이팅에만 사용하면 총 비용을 1만원 이하로 충분히 억제할 수 있다. ([OpenAI Developers][4])

## 6. 프롬프트 최적화 범위

이번 프로젝트는 “무한한 자유 프롬프트 탐색”을 하지 않는다. GLM-OCR의 공식 문서는 document parsing 쪽에서 `Text Recognition:` 같은 태스크 프롬프트를 제시하고, Prompt Learning은 dataset 기반의 반복 개선 루프를 권장한다. 따라서 탐색 공간은 다음처럼 제한한다. ([Hugging Face][5])

탐색 대상은 네 축이다.

첫째, **전사 강도**다.
예: “보이는 글자를 그대로 전사하라”, “번역하지 마라”, “추정하지 마라”.

둘째, **언어 제약**이다.
예: “출력은 한국어 텍스트 기준으로 유지하라”, “중국어로 바꾸지 마라”.

셋째, **불확실성 처리**다.
예: “확실하지 않은 글자는 원문 형태를 최대한 보존하라”, “추측 보정하지 마라”.

넷째, **반복 억제**다.
예: “같은 문자열을 반복 생성하지 마라”, “인식 불가 시 반복 대신 중단하라”.

즉, prompt optimizer가 바꾸게 할 것은 문학적 문체가 아니라, **OCR 전사 규칙**이다.

## 7. 선택할 Optimization 방식

Arize 공식 자료에는 Prompt Learning SDK와 Phoenix cookbook이 있으며, Prompt Optimization Techniques 문서는 few-shot, meta prompting, prompt gradients, DSPy prompt tuning을 비교한다. 하지만 이번 프로젝트는 예산이 매우 작고, task model이 로컬 GLM-OCR이며, optimizer model이 gpt-5-nano인 상황이다. 이 조건에서는 **Meta Prompting 기반의 경량 Prompt Learning 방식**이 가장 실용적이다. ([Arize AI][3])

따라서 본 PRD는 다음 방식을 채택한다.

1. baseline 프롬프트로 OCR 수행
2. 오답 사례와 정답을 비교
3. 오류 유형을 자동 요약
4. gpt-5-nano에게 “현재 프롬프트, 대표 실패 사례, 원하는 목표”를 주고 새 프롬프트 후보 생성
5. 후보 프롬프트를 다시 GLM-OCR에 적용
6. CER와 안정성 패널티를 다시 계산
7. 상위 후보만 남겨 2~3 라운드 반복

DSPy나 prompt gradients는 흥미롭지만, 이번 목표는 연구용 프레임워크 비교가 아니라 **적은 돈으로 실제로 먹히는 프롬프트를 뽑는 것**이므로 제외한다.

## 8. 평가 설계

평가 함수는 아래처럼 정의한다.

기본 점수는 CER 기반이다.

[
Score_{base} = 1 - CER
]

여기에 안정성 패널티를 적용한다.

[
Score = (1 - CER) - \alpha \cdot H - \beta \cdot R - \gamma \cdot E
]

여기서
(H) 는 비한국어 혼입 패널티,
(R) 는 반복 생성 패널티,
(E) 는 공백/의미 없는 출력 패널티다.

초기 권장값은 다음과 같다.

* (\alpha = 0.10)
* (\beta = 0.15)
* (\gamma = 0.15)

이렇게 두는 이유는, 실제 운영에서 “조금 틀린 한국어”보다 “중국어로 새어버리거나 반복 폭주하는 출력”이 더 치명적이기 때문이다.

세부 판정 규칙은 다음과 같다.

비한국어 혼입은 Unicode script 기준으로 한글 비율이 지나치게 낮거나, CJK 중 한자 비중이 비정상적으로 높을 때 패널티를 부여한다.
반복 생성은 동일 n-gram이 10회 이상 반복되면 패널티를 부여한다.
공백/무의미 출력은 길이가 너무 짧거나 의미 문자 비율이 낮을 때 패널티를 부여한다.

## 9. 실험 절차

실험은 4단계로 진행한다.

### 9.1 Baseline 측정

KORIE 검증 샘플 100장에 대해 baseline `Text Recognition:`으로 OCR을 수행한다. 이 결과가 모든 비교의 기준선이다. KORIE는 OCR 전사 라벨을 제공하므로 CER 계산이 가능하다. ([MDPI][2])

### 9.2 개발셋 기반 Prompt Optimization

개발용 60장에 대해서만 optimizer loop를 돌린다. 각 라운드는 다음 순서다.

* 현재 프롬프트로 GLM-OCR 실행
* CER 및 패널티 계산
* 상위 실패 케이스 10~15건 추출
* gpt-5-nano로 개선 프롬프트 3~5개 생성
* 새 프롬프트 재평가
* 최고 점수 후보만 다음 라운드로 전달

라운드는 최대 3회까지만 수행한다. 이유는 1660 Super에서 GLM-OCR 추론 시간이 짧지 않을 가능성이 크고, Prompt Learning 역시 iteration을 권장하지만 무한 반복을 전제로 하지는 않기 때문이다. ([Arize AI][3])

### 9.3 최종 검증

개발셋에서 고른 최종 프롬프트 1개와 baseline을 검증셋 100장에 동시에 평가한다. 이 단계에서는 optimizer를 더 돌리지 않는다.

### 9.4 산출물 고정

최종 산출물은 세 가지다.

* 최종 프롬프트 1개
* baseline 대비 개선 수치 표
* 대표 실패/개선 사례 10건

## 10. 프롬프트 후보 생성 정책

초기 seed prompt는 사람이 직접 4개만 준비한다.

* P0: `Text Recognition:`
* P1: 한국어 전사 지시 추가
* P2: 한국어 전사 + 번역 금지 + 추정 금지
* P3: 한국어 전사 + 반복 억제 + 불확실성 보수 처리

이후부터는 gpt-5-nano가 후보를 생성한다. Arize Prompt Learning 문서도 기본적으로 “현재 프롬프트 + 데이터셋 + 평가 기준”을 주고 optimizer가 개선안을 제안하는 구조를 갖는다. ([Arize AI][3])

## 11. 비용 계획

예산 상한은 1만원이다.

비용의 대부분은 OpenAI optimizer 호출이고, OCR 추론은 로컬 GPU이므로 현금 비용이 없다. gpt-5-nano는 공식 가격상 매우 저렴하므로, 아래처럼 통제하면 예산을 넘기기 어렵다. ([OpenAI Developers][4])

* optimizer 라운드 최대 3회
* 라운드당 후보 생성 최대 5개
* 후보 평가용 실패 사례 요약 텍스트는 15건 이하로 제한
* 이미지 자체를 optimizer에 보내지 않고, **오답/정답 텍스트 쌍과 오류 요약만 보냄**
* 검증셋은 최종 1회만 사용

이 설계면 OpenAI 사용량은 주로 텍스트 토큰에 머물고, 비용은 매우 작다.

## 12. 구현 범위

필수 구현물은 다음과 같다.

첫째, **GLM-OCR 실행기**다. 입력 이미지와 프롬프트를 받아 plain text를 저장한다.
둘째, **평가기**다. CER, 비한국어 혼입, 반복 생성, 공백/무의미 출력을 계산한다.
셋째, **optimizer driver**다. 현재 프롬프트와 실패 사례를 받아 새 프롬프트 후보를 만든다.
넷째, **experiment logger**다. 각 프롬프트별 점수와 샘플 결과를 CSV 또는 JSONL로 저장한다.

Arize Phoenix는 실험 추적과 prompt comparison에 도움을 줄 수 있지만, 이번 PRD에서는 **필수 의존성으로 두지 않는다**. 이유는 Prompt Learning SDK가 OpenAI integration을 공식 안내하고 있긴 하지만, 현재 프로젝트의 핵심은 “Arize 플랫폼 활용” 자체가 아니라 “Arize 방식의 optimization loop를 저비용으로 재현”하는 데 있기 때문이다. ([Arize AI][3])

## 13. 리스크와 대응

가장 큰 리스크는 **프롬프트보다 이미지 품질 영향이 더 큰 경우**다. KORIE는 원래 열화된 영수증을 포함하므로, prompt optimization만으로 대폭 개선되지 않을 수 있다. 논문도 한국어 영수증 OCR이 열악한 인쇄 품질과 mixed scripts 때문에 어렵다고 설명한다. ([MDPI][2])

두 번째 리스크는 **GLM-OCR가 문서 파싱 태스크용으로 설계된 모델이라, 자유 프롬프트 변화의 효과가 제한적일 수 있다는 점**이다. 이 경우에도 프로젝트는 실패가 아니다. “어떤 범주의 지시가 효과가 없었는지”가 명확히 남는다.

세 번째 리스크는 **optimizer overfitting**이다. 개발셋에서만 잘 되고 검증셋에서 무너질 수 있다. 그래서 개발셋과 검증셋을 반드시 분리한다.

## 14. 최종 의사결정 규칙

최종 프롬프트는 다음 규칙으로 채택한다.

1. 검증셋에서 baseline보다 CER가 더 좋아야 한다.
2. 비한국어 혼입 또는 반복 생성 중 적어도 하나는 유의미하게 줄어야 한다.
3. 프롬프트 길이가 지나치게 길어져 추론 안정성을 해치면 탈락한다.
4. 성능 차이가 미미하면 더 짧고 단순한 프롬프트를 선택한다.

즉, 최종 목표는 “가장 똑똑해 보이는 프롬프트”가 아니라 **가장 싸고, 단순하고, 덜 망가지는 프롬프트**다.

## 15. 제안 일정

1일차에는 데이터셋 준비와 baseline 측정을 끝낸다.
2일차에는 평가기와 패널티 룰을 구현한다.
